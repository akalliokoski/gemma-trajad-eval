# Project specification

**Gemma-TrajAD: Local trajectory anomaly detection for tool-using agents with Gemma 4 E2B**

---

## High-level objective

Build a local research/dev project that:

- Converts Hermes-style agent traces into a TrajAD-like supervised dataset
- Fine-tunes **Gemma 4 E2B-it** to:
  - Detect whether a trajectory is anomalous
  - Localize the bad step
  - Classify the anomaly type
- Integrates with an agent tracing stack such as **Langfuse**, **Phoenix/OpenInference**, or a lightweight local agent demo

---

## Why this direction is coherent

TrajAD frames the task as **trajectory anomaly detection** plus **step-level localization**, with anomaly types including **Task Failure**, **Process Inefficiency**, and **Unwarranted Continuation**. It also introduces **TrajBench**, which is generated by perturbing expert trajectories from **AgentBank**.

Hermes traces are a strong substrate for a similar pipeline because they already encode the intermediate reasoning and tool interactions needed for trajectory-level auditing.

Gemma 4 E2B is a good local model choice because Google positions the E2B/E4B line for edge and laptop deployment, with native reasoning, long context, and structured tool use. The E2B model card lists native system prompts, function calling, and 128K context for the small models.

## Subproject A — Hermes traces → TrajAD-style supervised dataset

### Goal

Create a reproducible dataset-generation pipeline that converts raw Hermes trajectories into labeled examples suitable for anomaly detection and step localization.

### Inputs

Primary source:

- **Hermes filtered traces dataset**
  `https://huggingface.co/datasets/DJLougen/hermes-agent-traces-filtered`
  Dataset card confirms ~3,679 rows, multi-turn trajectories, and ShareGPT-compatible format with tool calls and reasoning.

Optional source:

- **Original Hermes reasoning traces**
  `https://huggingface.co/datasets/lambda/hermes-agent-reasoning-traces`
  Larger but noisier source.

Reference structure:

- **Hermes trajectory format docs**
  `https://hermes-agent.nousresearch.com/docs/developer-guide/trajectory-format`
- **Hermes batch processing docs**
  `https://hermes-agent.nousresearch.com/docs/user-guide/features/batch-processing`

### Output dataset tasks

Create three supervised tasks from the same underlying trace:

**Task 1: Binary anomaly detection**

```
Input:  full trajectory
Output: {"anomalous": true}
```

**Task 2: Step localization**

```
Input:  full trajectory
Output: {"bad_step": 5}
```

**Task 3: Typed anomaly classification**

```
Input:  full trajectory
Output: {"anomaly_type": "process_inefficiency"}
```

**Task 4: Joint output (optional)**

```
Input:  full trajectory
Output: {
  "anomalous": true,
  "bad_step": 5,
  "anomaly_type": "wrong_tool_choice",
  "explanation": "The agent selected search_wikipedia when internet search was needed."
}
```

### Anomaly taxonomy

Top-level classes (from TrajAD framing):

| Class | Description |
|-------|-------------|
| `normal` | Valid expert trajectory with no anomalous steps |
| `task_failure` | Agent failed to complete the assigned task |
| `process_inefficiency` | Task completed but via a suboptimal path |
| `unwarranted_continuation` | Agent continued after task was effectively solved |

Operational subtypes for Hermes traces:

| Subtype | Top-level class | Description |
|---------|----------------|-------------|
| `wrong_tool_choice` | process_inefficiency | Agent selected a semantically inappropriate tool |
| `bad_tool_arguments` | task_failure | Agent passed malformed or incorrect arguments |
| `skipped_required_step` | task_failure | Agent omitted a necessary tool call or reasoning step |
| `repeated_step` | process_inefficiency | Agent duplicated an earlier tool call without new information |
| `premature_final_answer` | task_failure | Agent produced a final answer before gathering required evidence |
| `continued_after_sufficient_evidence` | unwarranted_continuation | Agent kept working after the task was effectively solved |
| `contradicted_tool_result` | task_failure | Agent's conclusion contradicts the actual tool response |
| `hallucinated_tool` | task_failure | Agent called a tool that does not exist in the provided schema |
| `invalid_tool_json` | task_failure | Agent produced malformed JSON in a tool call |
| `unnecessary_replanning` | process_inefficiency | Agent restarted planning without a valid reason |

### Dataset generation strategy

#### Stage A1 — Clean "normal" set

Take filtered Hermes traces as the gold set. The filtered dataset specifically retains trajectories with stronger reasoning depth, validation, self-correction, and valid JSON tool use.

#### Stage A2 — Synthetic perturbations

Generate anomalous variants by perturbing normal trajectories.

Perturbation rules:

| Rule ID | Description | Resulting subtype |
|---------|-------------|-------------------|
| `P1` | Replace selected tool with a semantically nearby wrong tool | `wrong_tool_choice` |
| `P2` | Corrupt one argument in a tool call | `bad_tool_arguments` |
| `P3` | Remove one necessary assistant/tool interaction pair | `skipped_required_step` |
| `P4` | Duplicate a tool call and response | `repeated_step` |
| `P5` | Append an unnecessary follow-up step after task is effectively solved | `continued_after_sufficient_evidence` |
| `P6` | Alter final answer so it contradicts the last tool result | `contradicted_tool_result` |
| `P7` | Truncate reasoning before the crucial decision | `premature_final_answer` |
| `P8` | Swap order of two dependent steps | `skipped_required_step` / `task_failure` |

#### Stage A3 — Automatic labeling

For every perturbed sample, record:

```json
{
  "source_trace_id": "uuid",
  "variant_id": "trace_000123_var_02",
  "is_anomalous": true,
  "anomaly_type": "bad_tool_arguments",
  "bad_step": 4,
  "generation_rule": "P2"
}
```

#### Stage A4 — Manual review set

Hand-review 100–200 samples to ensure:

- Anomaly is realistic and would confuse a capable agent
- Localization is correct (step index matches the perturbed step)
- Labels are consistent with the taxonomy

### Dataset format

JSONL with one example per record:

```json
{
  "id": "trace_000123_var_02",
  "source_trace_id": "uuid",
  "split": "train",
  "trajectory": [
    {"role": "system",    "content": "..."},
    {"role": "user",      "content": "..."},
    {"role": "assistant", "content": "<think>...</think><tool_call>...</tool_call>"},
    {"role": "tool",      "content": "<tool_response>...</tool_response>"}
  ],
  "is_anomalous": true,
  "anomaly_type": "bad_tool_arguments",
  "bad_step": 4,
  "generation_rule": "mutate_argument_value",
  "metadata": {
    "category": "Terminal & Coding",
    "subcategory": "Terminal Tasks"
  }
}
```

### Deliverables for subproject A

- `dataset_builder/` — full pipeline scripts
- Perturbation scripts with reproducible seeds
- Label schema (JSON Schema)
- Reviewed seed set (100–200 hand-checked samples)
- Hugging Face dataset publication
- Short data card documenting perturbation rules

## Subproject B — Gemma 4 E2B anomaly detector

### Goal

Fine-tune **Gemma 4 E2B-it** as a local verifier/judge for agent trajectories.

### Model selection

| Model | HF ID | Rationale |
|-------|-------|-----------|
| Primary | [`google/gemma-4-E2B-it`](https://huggingface.co/google/gemma-4-E2B-it) | Lower memory footprint; reasoning + structured output + 128K context |
| Optional | `google/gemma-4-E4B-it` | Comparison target for capability vs. memory tradeoff |

Why E2B first:

- Lower memory footprint for 32 GB M3
- Still includes reasoning, structured output, and long context
- Better fit for first local LoRA run
- Allows fast iteration before committing to larger models

### Training objective

Use SFT for structured judging output.

Prompt structure:

```
[SYSTEM] You are a trajectory anomaly detector. Given an agent trajectory, 
         identify whether it contains an anomaly, locate the bad step, 
         and classify the anomaly type.

[USER]   Trajectory:
         <trajectory JSON>
         
         Task: <binary|localize|joint>

[ASSISTANT] <structured JSON judgment>
```

Example judgment target:

```json
{
  "anomalous": true,
  "bad_step": 5,
  "anomaly_type": "skipped_required_step",
  "confidence": 0.81,
  "explanation": "The agent answered before checking the required file contents."
}
```

### Training runs

Run sequence:

| Run | Objective | Notes |
|-----|-----------|-------|
| 1 | Binary anomaly detection only | Simplest, easiest to debug |
| 2 | Binary + localization | Adds step-index prediction |
| 3 | Joint anomaly + localization + type | Full task |

This staged approach reduces debugging complexity. Validate each run against held-out set before proceeding.

### Fine-tuning approach

| Method | Stage | Rationale |
|--------|-------|-----------|
| LoRA SFT | Stage 1 (Mac) | Highest ROI; simple; well-supported in mlx-tune and Unsloth |
| QLoRA SFT | Stage 1 fallback | Use if memory limits are hit |
| DPO | Stage 2 (Mac) | Improves ranking between competing judgments; requires chosen/rejected pairs |
| GRPO | Stage 3 (cloud) | Reward-driven optimization; only after stable SFT and reward design |
| Full fine-tuning | Not recommended | Too costly; adapters are sufficient |

### Hardware plan

| Environment | Model | Method |
|-------------|-------|--------|
| Apple M3 / mlx-tune | Gemma 4 E2B | LoRA SFT, then DPO |
| Cloud GPU (A100) | Gemma 4 E4B / 31B | LoRA SFT, then DPO, then GRPO |

### Metrics

- Anomaly classification accuracy (binary)
- Macro F1 for anomaly type (10-class)
- Exact-match bad-step accuracy
- ±1 tolerance localization accuracy
- JSON validity rate
- Calibration / confidence sanity checks (expected calibration error)
- Inference latency on local Mac

### Success criteria

| Level | Criteria |
|-------|---------|
| Good MVP | Binary detection > base model; localization useful on obvious perturbations; JSON validity > 95%; stable local inference |
| Strong result | Useful classification on 5–8 anomaly subtypes; demo integration with observability stack |

## Milestones

### Week 1
- Load and inspect Hermes filtered traces
- Write trajectory normalizer
- Define anomaly taxonomy
- Review 30 traces manually

### Week 2
- Implement first 4 perturbation rules (P1–P4)
- Generate first synthetic anomalous dataset
- Hand-check 100 examples
- Publish dataset schema draft

### Week 3
- Create SFT prompts for binary detection
- Fine-tune Gemma 4 E2B on small sample
- Evaluate on held-out synthetic set

### Week 4
- Add localization (Run 2)
- Build local demo with one tracing tool (Langfuse or Phoenix)
- Publish repo and write-up

### Week 5+
- Compare E2B vs E4B
- Add harder anomaly types (P5–P8)
- Release Hugging Face dataset and adapter

## MVP decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Model | Gemma 4 E2B-it | Fits M3 32GB; fast iteration |
| Data source | DJLougen/hermes-agent-traces-filtered | Clean, filtered, ready to use |
| Dataset subproject | Yes, first-class | Publishable artifact; makes fine-tuning reproducible |
| Tasks | Binary + localization | Simple enough to finish; substantial enough to share |
| Perturbation rules | 4 rules for MVP | P1–P4 cover the most common failure modes |
| Eval set | 150 hand-checked samples | Manageable; sufficient for early signal |
| Community integration | Langfuse or Phoenix first | Most natural fit for trace-level scoring |

## Community / contribution targets

### Langfuse
- **Repo:** https://github.com/langfuse/langfuse
- **Why:** Open-source LLM tracing / evaluation platform. Natural place to demo trajectory anomaly scoring over stored agent traces.
- **Potential contribution:** Cookbook — "trajectory anomaly detection for agent traces"; evaluator integration that scores a trace and flags the suspicious step.

### Arize Phoenix
- **Repo:** https://github.com/Arize-ai/phoenix
- **Why:** Open-source AI observability. Strong fit for trace-level anomaly detection and visualization.
- **Potential contribution:** Notebook or plugin-like eval example for localized trajectory failure analysis.

### OpenInference
- **Spec:** https://arize-ai.github.io/openinference/spec/
- **Why:** Standardized trace schema for LLM/agent/tool/retrieval workflows. Excellent place to align dataset and outputs with community conventions.
- **Potential contribution:** Proposal or example fields for `trajectory.anomalous`, `trajectory.bad_step`, `trajectory.anomaly_type`, `trajectory.confidence`.

### AgentOps
- **Repo:** https://github.com/AgentOps-AI/agentops
- **Why:** Agent monitoring and failure detection orientation. The detector can act as an offline or runtime trace-quality scorer.

### smolagents
- **Docs:** https://huggingface.co/docs/smolagents/en/tutorials/inspect_runs
- **Why:** Easy place to create local demo traces. Straightforward demo path: run agent → export trace → pass to Gemma verifier → show flagged step.
